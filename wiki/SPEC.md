## 1. General Architecture

```
<workspace>/
├── AGENTS.md                  ← operational instructions (to be updated with wiki ref)
├── SOUL.md                    ← Agent persona (unchanged)
├── IDENTITY.md                ← name, tone (unchanged)
├── skills/
│   └── wiki-core.md           ← skill loaded at every OpenClaw session
├── wiki-session.md            ← current session state (generated by wiki.py)
├── wiki.config.json           ← instance configuration
├── scripts/
│   ├── wiki.py                ← single entry point
│   ├── wiki_embed.py          ← chunking + bge-m3 embedding
│   ├── wiki_lancedb.py        ← LanceDB operations
│   └── wiki_index.py          ← index.md generation
├── wiki/                      ← Level 1: permanent knowledge
│   ├── .schema.md
│   ├── log.md
│   ├── entities/
│   ├── concepts/
│   └── synthesis/
├── wiki-works/                ← Level 2: active research
│   └── <project>/
│       ├── .schema.md
│       ├── log.md
│       ├── raw/
│       ├── entities/
│       ├── concepts/
│       └── synthesis/
└── memory/
    └── lancedb/               ← excluded from git, rebuildable
```

**Core invariant:** Agent never writes directly to the wiki. Everything goes through `wiki.py`. The skill governs the *when* and *why*; the scripts handle the *how*.

## 2. OpenClaw Integration

### Skill loading

`wiki-core.md` lives in `<workspace>/skills/` — OpenClaw loads it automatically at every session according to the skill precedence hierarchy.

`wiki-session.md` is not a traditional skill: it is a file that Agent actively reads. Add to `AGENTS.md`:

```
At the start of each session read <workspace>/wiki-session.md for the current wiki context.
Before any wiki operation, re-read skills/wiki-core.md to verify the protocol.
```

The second line forces Agent to reload the rules before acting — not only at session start — mitigating instruction weight loss over long context windows.

### LanceDB separate from personal memory

The wiki uses `memory/lancedb/` **physically separate** from Agent's existing hybrid RAG memory. The two systems share neither directories nor tables. Agent is the integration point: during QUERY it interrogates both systems and synthesises the response without explicit distinction between sources.

**Synthesis rule:** a wiki page is created only if the threshold criteria from DESIGN.md (§synthesis-threshold) are met using wiki sources. The contribution of personal memory to the response does not count toward the creation of new pages.

## 3. `wiki-core.md` — Permanent Skill

### Structure

```markdown
---
name: wiki-core
description: Wiki protocol for AI Agent — intent classification, workflows, checklists
---

[Mandatory checklist, intent classification table, workspace selection,
classification output — as per §3 of this spec]
```

The actual `wiki-core.md` file will contain the full text of the following sections,
formatted as direct instructions to Agent (second person singular).

### Mandatory pre-action checklist

Agent executes this checklist for every message before proceeding:

```
1. Read wiki-session.md → check status (ok / in-progress / needs-repair)
2. If status ≠ ok → notify the user before any operation
3. Classify the intent of the message (see §classification)
4. Does the message contain more than one intent? If so, handle them in sequence
5. Emit classification line: [INTENT: X | WORKSPACE: Y | CERTAINTY: high/medium/low]
6. If CERTAINTY = low → ask user for confirmation before proceeding
7. Execute the workflow corresponding to the intent
```

### Intent classification

| Linguistic pattern | Intent |
|--------------------|--------|
| "study this", "save", "I found", "read this", bare URL, attachment | INGEST |
| direct question, "what do you know about", "tell me", "explain", "how does it work" | QUERY |
| "check the wiki", "cleanup", "lint", "maintenance" | LINT |
| everything else | AMBIGUOUS → ask for confirmation |

**Confirmation protocol (ambiguous):** a single line, never verbose:
> "Do you want me to save this to the wiki, or are you just sharing it?"

### Automatic workspace selection

1. Read `wiki.config.json` → list of projects with keywords
2. Count matches between message keywords and each project's keywords
3. If one project has clearly more matches → select it
4. If two projects are tied → ask the user
5. If no matches → use the main `wiki/`

### Classification output (always visible before acting)

```
[INTENT: INGEST | WORKSPACE: trading | CERTAINTY: high]
```

The user can correct this line before Agent executes any operation.

## 4. `wiki-session.md` — Session State

Generated exclusively by `wiki.py session-update`. Agent never modifies it directly.

```markdown
# Wiki Session — YYYY-MM-DD HH:MM

## Status
status: ok | in-progress | needs-repair

## Active Workspace
Project: <name>
Path: <relative path>

## Last Operation
Type: ingest | query | lint | rebuild | none
Completed: YYYY-MM-DD HH:MM
Detail: <summary output>

## Main Wiki
Total pages: N | Last lint: YYYY-MM-DD
```

**Robustness:** the `status: in-progress` field is written at the start of every operation; `status: ok` only upon completion. A crash leaves `in-progress` → Agent detects this at the next session and notifies the user.

## 5. `wiki.py` — Entry Point

### Interface

```
wiki.py <command> [arguments]

  ingest         --workspace <path> --source <url|file> --title <str>
  query          --workspace <path> --q <string>
  lint           --workspace <path> [--full]
  index          --workspace <path>
  rebuild        --workspace <path>
  session-update --workspace <path> --op <type> --status <ok|failed|in-progress> [--detail <json>]
```

### JSON output

Every command produces JSON on stdout:

```json
{ "status": "ok", "op": "ingest", "pages_written": 3, "conflicts": [], "mini_lint": "ok" }
{ "status": "error", "code": "checkpoint_failed", "message": "...", "recoverable": true }
{ "status": "conflict", "level": 3, "page": "...", "detail": "..." }
```

Agent reads this output and presents it to the user in natural language.

### Lock file

At startup each command writes `<workspace>/.wiki-lock`. It removes it upon completion (ok or error).

If `.wiki-lock` exists at startup → output:
```json
{ "status": "error", "code": "lock_exists", "message": "Previous operation did not complete", "recoverable": true }
```
Agent notifies the user and does not proceed.

### Config validation

`wiki.py` validates `wiki.config.json` at the startup of every command. If a required field is missing:
```json
{ "status": "error", "code": "invalid_config", "missing_field": "thresholds.staleness_days" }
```

## 6. Python Modules

### `wiki_embed.py`

- Uses the bge-m3 native tokenizer to count tokens (resolves §chunking ambiguity)
- Boundary-aware chunking: never cuts in the middle of `##` or `###` sections
- Chunk size: 512 tokens, overlap: 64 tokens
- Whole-page vs chunked threshold: 1500 tokens (bge-m3 tokenizer)
- Output: `List[{chunk_id, chunk_text, vector, content_hash, page_hash}]`

### `wiki_lancedb.py`

Tables: `wiki_pages`, `staging_wiki_pages` (identical schema).

Schema:
```
path          string   — relative path from workspace root
chunk_id      int      — 0 = whole page, 1..N = chunk
chunk_text    string   — embedded text
content_hash  string   — SHA256(chunk_text)
page_hash     string   — SHA256(entire page) for detecting rechunking
vector        float[1024]
last_embedded float    — unix timestamp
```

Index: `(path, chunk_id) UNIQUE`, `vector ANN (HNSW)`.

**Correct upsert:** `upsert(path, chunks)` deletes *all* chunks with that path, then inserts the new ones. This eliminates orphan chunks when a page changes its chunk count.

**Detect rename:** compare `set(path in LanceDB)` vs `set(file on filesystem)`. If a LanceDB-only path and a filesystem-only path share the same `content_hash` → rename → update path without re-embedding.

**Atomic operations:**
- `promote_staging()` → upsert from staging to wiki_pages, clears staging
- `rollback_staging()` → clears staging, no changes to wiki_pages

### `wiki_index.py`

- "Stale" defined as: `mtime(index.md) < max(mtime(*.md))` in the wiki directory
- Reads `index_token_budget` from `wiki.config.json`
- Three-level reduction strategy if budget is exceeded:
  1. Remove descriptions → only `- [[slug]]`
  2. Separate indices per category
  3. Alphabetical split for very large categories

## 8. §error-states — Invalid States and Recovery

| State | Signal | Automatic action | User intervention |
|-------|--------|------------------|-------------------|
| `.wiki-lock` present | Lock file exists | Agent notifies, does not proceed | Confirm whether to proceed |
| `.tmp` on disk without lock | Ingest interrupted | `wiki.py` deletes `.tmp`, rolls back staging, logs `ingest-failed` | No |
| `wiki-session.md` status `in-progress` | Previous session did not complete | Agent notifies before any operation | Confirm recovery or reset |
| LanceDB missing or corrupt | DB open error | Proposes `wiki.py rebuild` | Confirm (destructive operation) |
| Config missing or malformed | Validation error | Agent notifies with specific field | Fix config |
| `staging_wiki_pages` non-empty at startup | Residual staging | `wiki.py` clears silently, log entry | No |
| Mini-lint failed | `mini_lint: failed` in JSON | Agent notifies, `wiki-session.md` → `needs-repair` | Run `wiki.py lint` |
| Level 3 conflict | `conflicts: [{level: 3}]` | Agent presents conflict, blocks merge of specific page | Decides canonical version |
| Skill `wiki-core.md` not loaded | Agent does not emit `[INTENT:]` line | — | Verify `<workspace>/skills/` |

**Principle:** `wiki.py` never leaves the system in a silently corrupt state. Every anomaly produces a specific JSON error code. Agent does not invent recovery steps — it follows this table or asks the user.

## 10. Consistency Rules with DESIGN.md v2

This spec **extends** DESIGN.md v2, it does not replace it. The sections of DESIGN.md remain valid. The sections present here take precedence in case of conflict.

Specific corrections to DESIGN.md v2 incorporated in this spec:
- §chunking: "token" = bge-m3 tokenizer (not character approximation)
- §index-generation: "stale" defined with mtime comparison
- §lancedb-schema: upsert operates on all chunks of the path, not only chunk_id=0
- §mini-lint: added `status` field in wiki-session.md for crash detection
- Log: added types `rebuild-lancedb`, `promote`, `rename`, `session-repair`
- Level 3 conflict: the partial block occurs by holding the page in staging (not promoted) until the user responds
